# Contributing

Thanks for considering a contribution. This repo stays useful only if the data in it is current, so contributions that verify or update existing programs are just as valuable as additions.

## Types of Contributions Welcome

- **Updates to existing programs.** Amounts, eligibility rules, and application URLs change. If you spot a stale entry, open a PR with the current information and a source link.
- **New programs.** Must meet the criteria in [How to Submit a New Program](#how-to-submit-a-new-program) below.
- **Corrections.** Typos, broken links, outdated claims.
- **Regional programs.** Non-US programs (EU, UK, India, LATAM, APAC) are welcome as long as eligibility is clear.

## What Gets Rejected

- Affiliate/referral links. The goal is to route founders to the primary application, not to a middleman.
- Paid aggregator services or subscription-gated "perk marketplaces." If the program requires someone to buy a third-party subscription to access the deal, it does not belong here.
- Programs with credit amounts under $1,000 in value unless the program is strategically meaningful (e.g., a credibility-building accelerator).
- Unverified claims. If you cannot link to the provider's own application page with the current terms, the entry will not be merged.
- Programs that require an NDA or non-disparagement clause to access benefits.

## How to Submit a New Program

Open a pull request adding the program under the correct section of `README.md`. Include:

1. **Program name** and direct application URL (the provider's own page, not an aggregator).
2. **Current credit amount or benefit value.** If the program has tiers, list each tier with its value.
3. **Eligibility requirements.** Be specific: funding stage, company age, team size, geographic restrictions, excluded business types.
4. **Source link** dated within the last 6 months proving the amount and terms.
5. **One-sentence context** explaining why the program is worth applying (what makes it distinct from similar programs in the same category).

Follow the formatting conventions of existing entries. Use inline code formatting for URLs, filenames, and paths. Avoid marketing language — this is a reference document, not a pitch.

## Verification Standards

Entries must be based on the provider's own documentation. Acceptable primary sources:

- The provider's official startup program landing page.
- Provider-published blog posts or announcements dated within the last 12 months.
- Official FAQ pages hosted on the provider's domain.

Not acceptable as sole sources:

- Third-party "awesome lists" (use them for discovery, then verify against the provider).
- LinkedIn posts or threads.
- Paid newsletter content behind a paywall.
- Reseller or consulting firm landing pages.

## Updating a Program Whose Terms Have Changed

If you know a program's terms have changed but don't have time to fully rewrite the entry, open an issue with the title `[UPDATE] Program Name — current terms` and link to the provider's updated page. Someone else can then do the rewrite.

## Removing a Program

Open a PR removing the entry with a clear commit message explaining why: program ended, terms became punitive, provider shut down, or eligibility narrowed to the point that the program is no longer useful to typical early-stage startups.

## Code of Conduct

Be decent. Disagreements about whether a program qualifies should stay focused on the merits, not on the people submitting entries.
